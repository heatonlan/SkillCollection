# auto-dev-tree — 自主递归开发框架

让 AI 像一棵树一样自主展开开发任务。你只需要提供根节点目标，AI 自动分解、实现、测试、验证，逐层完成。

## 核心理念

传统的 AI 辅助开发需要人类手动编排：拆任务 → 逐个指示 AI 开发 → 手动测试 → 发现问题 → 再指示修复。

**Auto Dev Tree** 将这个过程自动化为一棵自展开的任务树：

```
        用户目标 (根节点)
        ┌───┴───┐
      子任务A   子任务B          ← 复杂度高，自动分解
      ┌─┴─┐      │
    A.1   A.2    B.1            ← 足够简单，直接执行 DTV
    DTV   DTV    DTV
     ↓     ↓      ↓
    ✓     ✓      ✓             ← 叶子节点验证通过
      └─┬─┘      │
     集成验证   集成验证         ← 父节点做集成验证
        └───┬───┘
        最终验收                 ← 根节点全量验证
```

## DTV 循环

每个任务节点都经过 **Develop → Test → Verify** 循环：

| 阶段 | 动作 | 失败处理 |
|------|------|---------|
| **D** Develop | 编写/修改代码 | — |
| **T** Test | 编写测试或验证脚本 | — |
| **V** Verify | 运行测试，检查结果 | 回到 D，最多重试 3 次 |

验证通过 → 节点完成 → 触发父节点集成验证

## 使用方式

在 Claude Code 中调用：

```
/auto-dev-tree 实现一个用户认证系统，包括注册、登录、JWT token 管理
```

AI 会自动：
1. 分析项目现状和目标
2. 分解为任务树（如：数据模型 → API 路由 → 中间件 → 前端表单）
3. 从叶子节点开始，逐个执行 DTV 循环
4. 逐层集成验证
5. 输出完成报告

## 复杂度判定

节点是否需要进一步分解的判定标准：

| 指标 | 分解阈值 | 说明 |
|------|---------|------|
| 涉及文件数 | > 3 个 | 单个原子任务不应跨太多文件 |
| 关注点层次 | > 2 层 | 如同时涉及 UI + API + DB |
| 预估变更行数 | > 80 行 | 过大的变更难以一次验证 |
| 可独立测试单元 | > 2 个 | 可独立测试的部分应拆开 |

分解原则：
- 2-4 个子节点为宜，过多过少都不好
- 子节点间依赖关系明确
- 每个子节点独立可验证

## 状态跟踪

通过 `goal-tree.json` 跟踪任务树状态：

```json
{
  "goal": "实现用户认证系统",
  "status": "in_progress",
  "maxRetries": 3,
  "nodes": {
    "root": {
      "id": "root",
      "goal": "用户认证系统",
      "status": "decomposed",
      "depth": 0,
      "children": ["auth-model", "auth-api", "auth-middleware"],
      "acceptanceCriteria": "注册、登录、JWT 刷新全流程可用"
    },
    "auth-model": {
      "id": "auth-model",
      "goal": "User 数据模型和密码哈希",
      "status": "done",
      "depth": 1,
      "parent": "root",
      "files": ["src/models/user.ts"],
      "verifyResult": { "pass": true, "detail": "模型测试通过" }
    }
  }
}
```

节点状态流转：

```
pending → dev → test → verify → done
                         ↓ (失败)
                        dev (重试)
```

## 验证策略

框架根据项目环境自动选择验证手段：

| 项目特征 | 验证方式 |
|----------|---------|
| 有 `package.json` scripts.test | `npm test` |
| 有 `pytest.ini` / `setup.cfg` | `pytest` |
| 有 `go.mod` | `go test ./...` |
| 有 `.eslintrc` | ESLint 检查 |
| 有 `verify.md` | verify-viewer 框架 |
| 无以上设施 | 创建临时验证脚本 |

## 适用场景

- **功能开发**：新增完整功能模块
- **重构**：大范围代码重构，需要保证不破坏现有功能
- **Bug 修复**：复杂 bug 涉及多个文件的修复
- **项目初始化**：从零搭建项目骨架

## 与其他技能的关系

| 技能 | 关系 |
|------|------|
| verify-viewer | 可作为根节点验收阶段的验证手段 |
| nxbuild | 如果目标涉及构建，在验证阶段调用 |
| cloudflare-tunnel | 如果目标涉及部署，作为子节点任务 |

## 设计原理

### 为什么是树而不是线性列表？

线性列表假设任务之间没有层次关系，无法表达"先完成子任务再集成验证"的模式。树结构天然支持分治法——大问题拆小问题，小问题独立解决，再组合验证。

### 为什么每个节点都需要 DTV？

没有验证的开发就是盲写代码。在每个最小单元验证通过后再集成，比写完所有代码再一起测试更容易定位问题。错误暴露得越早，修复成本越低。

### 为什么限制重试次数？

避免 AI 在一个问题上陷入无限循环。3 次重试仍失败说明可能需要人类介入——也许需求不清晰，也许有环境问题，也许需要重新分解任务。

### 为什么用 JSON 跟踪状态？

Claude Code 的对话可能很长，JSON 文件作为外部状态存储，确保 AI 在长对话中不会丢失进度。同时也方便人类查看当前进度。
